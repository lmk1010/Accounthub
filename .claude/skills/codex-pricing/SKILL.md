# codex-pricing

## 用途

AccountHub 运营方需要频繁回答三类问题：
1. **给定利用率，最低定价是多少**（保本线）
2. **给定定价，需要多少利用率才保本**（反推 U）
3. **给定目标月利润，定价/利用率组合怎么选**

每次用户问这类问题，本技能按统一公式和当前基准参数作答，不要凭感觉估。

---

## 基准参数（2026-04-24 实测校准）

| 代号 | 含义 | 默认值 | 来源 |
|---|---|---|---|
| `C` | 单 Pro 账号月成本 (¥) | **160** | 灰产采购价 |
| `K` | 单 Pro 账号月容量 (客户 $) | **12,266** | rebekah 实测 12%/7d → 反推 100% ≈ $2,833/周 × 4.33 |
| `N` | Pro 账号数量 | **3** | 2026-04-24 时 |
| `U` | 目标利用率 | 视问题 | 通常 0.25 / 0.50 / 1.0 |
| `P` | 定价 (RMB/$1 quota) | 视问题 | 当前 0.15 |
| `M` | 模型混合系数 | **1.0**（全 5.4） | gpt-5.5 因 2× 更贵，若客户大量用 5.5 这里降到 0.5 |
| `S` | 安全系数 (考虑封号/退款) | **1.3** | 封号率、客服成本、退款缓冲 |

**当用户场景变了（成本降/涨、号数量变、客户大量用 5.5 等），先读最新参数再算。**

---

## 核心三公式

### 公式 1：保本定价（给 U，求最低 P）

```
P_min = (C × S) / (K × U × M)
```

用途：用户问"50% 利用率下最低一刀多少钱"就套这个。
默认 `S=1.3`（含安全裕度）。若用户明确"纯保本不留裕度"，传 `S=1.0`。

### 公式 2：保本利用率（给 P，求最低 U）

```
U_min = (C × S) / (K × P × M)
```

用途：用户问"我定 0.1 RMB/$1 需要用满多少才不亏"。

### 公式 3：目标利润反推（给目标月净利 G，求 P 或 U）

```
对固定 U：  P = (C + G/N) × S / (K × U × M)
对固定 P：  U = (C + G/N) × S / (K × P × M)
```

`G` = 月净利目标（¥），`N` = 账号数。

### 组合：N 个账号场景

月收入 = `N × K × U × P × M`
月成本 = `N × C`
**月净利 = `N × (K × U × P × M − C)`**

## 标准案例（用来验证公式对不对）

基准：C=160, K=12266, N=3, M=1, S=1.3

| 问题 | 公式代入 | 答案 |
|---|---|---|
| 50% 利用率最低定价 | P = (160×1.3)/(12266×0.5×1) | **¥0.0339/$1** |
| 50% 利用率纯保本（S=1） | P = 160/(12266×0.5) | **¥0.0261/$1** |
| 100% 利用率保本 | P = 160/12266 | **¥0.0130/$1** |
| 定价 0.15，保本利用率 | U = 160/(12266×0.15) | **8.7%** |
| 定价 0.10，保本利用率 | U = 160/(12266×0.10) | **13.0%** |
| 定价 0.05，保本利用率 | U = 160/(12266×0.05) | **26.1%** |
| 3 Pro 月净利目标 ¥5,000，定价 0.15，要多少利用率 | U = (160+5000/3)/(12266×0.15) | **100%**（天花板） |
| 3 Pro 月净利目标 ¥3,000，定价 0.10，要多少利用率 | U = (160+3000/3)/(12266×0.10) | **94.6%** |

## 答题结构（每次都按这个走）

1. **确认参数**：C、K、N 是否用默认，还是用户给了新值
2. **套公式**：用上面三式之一
3. **给两版答案**：
   - **纯保本**（S=1）：理论最低价，无安全边际
   - **安全保本**（S=1.3）：推荐日常使用
4. **配套解读**：
   - 对比当前 0.15 RMB/$1，新定价降了多少倍？
   - 降价后月利润变化（按真实当前利用率 ~12% 套）
5. **风险提示**：封号、模型结构变化（5.5 占比上升）、突发高销量导致超卖

## 计算脚本

已放在 `scripts/codex-pricing.py`，支持命令行传参快速算。默认跑一遍给用户看对比表。使用：

```bash
python3 scripts/codex-pricing.py --util 0.5                    # 50% 利用率下的保本价
python3 scripts/codex-pricing.py --price 0.15                  # 定价 0.15 需要多少利用率
python3 scripts/codex-pricing.py --util 0.5 --target-profit 3000  # 目标月利 3000 的组合
python3 scripts/codex-pricing.py --scan                        # 扫描常见场景，一表输出
```

默认 `C=160 K=12266 N=3 M=1 S=1.3`，可用 `--cost/--capacity/--accounts/--model-mix/--safety` 覆盖。

## 关键判断

- **"当前 0.15 亏不亏"**：保本利用率 8.7%，rebekah 实测 ~12%/周 = ~50%/月，**不亏且有裕度**。
- **"能不能降到 0.10 甚至 0.08"**：保本利用率 13~16.3%，当前水平覆盖得住；降价拉新空间大。
- **"1毛5望而却步"**：定价对标对手（大陆中间商 Codex 代充普遍 0.08-0.12/$1）确实偏高，可压到 0.10 做价格战。
- **什么情况下必须涨价**：
  - 客户模型占比里 gpt-5.5 >50%（M 跌到 0.5），同定价下利润腰斩
  - 号寿命 <15 天（有效 C 翻倍）
  - 号采购价上涨

## 反面警告

- **不要用这个 skill 做客户端销售预测**。本技能只算"单号/多号的成本容量比"；预测销量是另外的事，需要销售数据 + 转化率模型，别把两件事混一起。
- **别把"降价到保本线"当建议**。保本线只定义了底，真实决策要综合竞争、品牌定位、毛利率目标。推荐保留至少 ≥3× 于保本线的毛利率。
